Patrón de Diseño: State (Comportamiento)

UNEV

Propósito

Permite a la entidad OrdenServicio modificar su comportamiento cuando cambia su estado interno (Recibida ➔ Diagnóstico ➔ Presupuestada, etc.). Evita el uso de condicionales masivos.

Diseño decoupled (OCP & LSP)

Cumple con el Principio Abierto/Cerrado. Para agregar un nuevo estado (ej. "Espera de Repuestos") no modificamos la orden, solo creamos un nuevo archivo de estado concreto que hereda de la clase base abstracta.

EstadoRecibida.js
const EstadoOrden = require('./EstadoOrden'); const EstadoEnDiagnostico = require('./EstadoEnDiagnostico'); // Estado inicial de la orden class EstadoRecibida extends EstadoOrden { avanzar(orden) { // Transición encapsulada orden._cambiarEstado(new EstadoEnDiagnostico()); } nombre() { return 'RECIBIDA'; } } module.exports = EstadoRecibida;

Patrón de Diseño: Strategy (Comportamiento)

UNEV

Propósito

Define una familia de algoritmos, encapsula cada uno de ellos y los hace intercambiables. En nuestro sistema se utiliza en la CalculadoraOrden para aplicar diferentes estrategias de impuestos y descuentos dominicanos.

Encapsulamiento del ITBIS 18%

La lógica de recargos, impuestos de ley (ITBIS) y descuentos corporativos se procesa mediante estrategias dinámicas. Esto aísla el cálculo financiero del flujo operativo de la orden.

CalculadoraOrden.js
class CalculadoraOrden { constructor(estrategiaDescuento) { this.estrategiaDescuento = estrategiaDescuento; this.TASA_ITBIS = 0.18; // 18% República Dominicana } calcular(orden) { const subtotal = orden.servicios.reduce(...) + orden.repuestos.reduce(...); const descuento = this.estrategiaDescuento.aplicar(subtotal); const itbis = (subtotal - descuento) * this.TASA_ITBIS; return { subtotal, descuento, itbis, total: subtotal - descuento + itbis }; } }

Conclusiones & Calidad de Software

UNEV

Validación del Acoplamiento

Se configuró dependency-cruiser en el proyecto backend. Si una clase de dominio o aplicación importa algo de infraestructura, los tests automáticos fallan, garantizando la pureza del diseño.

Consistencia por Eventos

El uso de Domain Events (EstadoCambiadoEvent) permite un acoplamiento asíncrono para notificaciones y facturación. Se utiliza el patrón pull del agregado para asegurar que solo se disparen tras persistir con éxito en Postgres.

Lecciones Aprendidas

  • ✔️ Mantenibilidad: Añadir nuevos requerimientos sin tocar código existente.
  • ✔️ Testabilidad: Testear el negocio sin necesidad de levantar bases de datos.
  • ✔️ Seguridad: Ciclo de vida blindado frente a transiciones inválidas.
