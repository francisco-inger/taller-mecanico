📐 Tema 1 — Principios SOLID en la Lógica de Negocio

El sistema SIGEST implementa una Arquitectura Hexagonal con Domain-Driven Design (DDD). A continuación se muestra cómo cada principio SOLID se aplica en código real del proyecto, con evidencia de refactorización antes/después.

SRP Single Responsibility Principle — Responsabilidad Única

Regla: Cada clase debe tener una sola razón para cambiar.

Problema detectado: El controlador HTTP vehiculo.routes.js contenía directamente la validación de campos obligatorios, la transformación de datos (parseInt), y la persistencia. Tres razones para cambiar en una sola función.

❌ ANTES — vehiculo.routes.js
// Controlador HTTP con lógica de negocio mezclada
router.post('/', async (req, res) => {
  const { placa, marca, modelo, anio } = req.body;

  // ❌ Validación de negocio en el controlador
  if (!placa || !marca || !modelo) {
    return res.status(400)
      .json({ error: 'Campos obligatorios' });
  }

  // ❌ Transformación de datos aquí
  const anioInt = parseInt(anio);

  // ❌ Acceso directo a Prisma
  const vehiculo = await prisma.vehiculo
    .create({ data: { placa, marca,
      modelo, anio: anioInt } });
  res.json(vehiculo);
});
✅ DESPUÉS — CrearVehiculoUseCase.js
// Caso de uso con una sola responsabilidad
class CrearVehiculoUseCase {
  constructor(vehiculoRepo) {
    this.vehiculoRepo = vehiculoRepo;
  }

  async ejecutar(datos) {
    const { placa, marca, modelo, anio } = datos;
    if (!placa || !marca || !modelo)
      throw new Error('Obligatorios');

    const anioValido = parseInt(anio);
    return this.vehiculoRepo.crear({
      placa, marca, modelo, anio: anioValido
    });
  }
}
// Controlador ahora solo delega:
router.post('/', async (req, res) => {
  const v = await facade.crearVehiculo(req.body);
  res.json(v);
});
✅ Olor eliminado: God Method — la función del controlador hacía demasiadas cosas. Ahora el controlador tiene una sola responsabilidad (recibir HTTP y delegar), y el Use Case tiene otra (validar y crear).
OCP Open/Closed Principle — Abierto/Cerrado

Regla: Abierto a la extensión, cerrado a la modificación.

Problema detectado: Si la lógica de descuento fuera un if/else dentro de la calculadora, cada nuevo tipo de descuento requeriría modificar la clase existente.

❌ ANTES — Lógica con if/else
// Cálculo de descuento con if/else (violación OCP)
function calcularDescuento(subtotal, tipo) {
  if (tipo === 'frecuente') {
    return subtotal * 0.10;
  } else if (tipo === 'corporativo') {
    return subtotal * 0.15;
  } else if (tipo === 'promocion') {
    return subtotal * 0.05;
  }
  // ❌ Para agregar un nuevo descuento hay que
  // modificar esta función ya probada
  return 0;
}
✅ DESPUÉS — Patrón Strategy
// CalculadoraOrden — Contexto del Strategy
class CalculadoraOrden {
  constructor(estrategia = new SinDescuento()) {
    this._estrategia = estrategia;
  }
  calcularTotal(servicios, repuestos) {
    const subtotal = ...;
    const desc = this._estrategia.calcular(subtotal);
    return { subtotal, descuento: desc, ... };
  }
}
// Agregar nuevo descuento = crear clase nueva:
class DescuentoCorporativo {
  constructor(porcentaje) { this.porcentaje = porcentaje; }
  calcular(sub) { return sub * (this.porcentaje / 100); }
  descripcion() { return `Corporativo (${this.porcentaje}%)`; }
}
// ✅ CalculadoraOrden NO se modifica
✅ Olor eliminado: Cascada if/else — el patrón Strategy permite agregar nuevos descuentos (DescuentoClienteFrecuente, DescuentoCorporativo, DescuentoPromocion, SinDescuento) creando clases nuevas, sin tocar la calculadora ya probada.
LSP Liskov Substitution Principle — Sustitución de Liskov

Regla: Una subclase debe poder reemplazar a su clase base sin alterar el comportamiento esperado.

Problema detectado: Sin el patrón State, la lógica de transiciones de estado estaría dispersa en condicionales que verifican el estado actual antes de cada operación.

❌ ANTES — Condicionales dispersos
// Sin patrón State (violación LSP)
function avanzarEstado(orden) {
  if (orden.estado === 'RECIBIDA') {
    orden.estado = 'EN_DIAGNOSTICO';
  } else if (orden.estado === 'EN_DIAGNOSTICO') {
    orden.estado = 'PRESUPUESTADA';
  } else if (orden.estado === 'PRESUPUESTADA') {
    orden.estado = 'APROBADA';
  }
  // ❌ Cada nuevo estado requiere otro else-if
  // ❌ No hay forma de "sustituir" un estado
  // por otro sin cambiar esta función
}
✅ DESPUÉS — Patrón State + LSP
// Clase base abstracta
class EstadoOrden {
  avanzar(orden) { throw new Error('No permitido'); }
  rechazar(orden) { throw new Error('No permitido'); }
  nombre() { throw new Error('Implementar'); }
}
// Cada estado concreto es intercambiable (LSP)
class EstadoRecibida extends EstadoOrden {
  avanzar(orden) {
    orden._cambiarEstado(new EstadoEnDiagnostico());
  }
  nombre() { return 'RECIBIDA'; }
}
class EstadoPresupuestada extends EstadoOrden {
  avanzar(orden) { orden._cambiarEstado(new EstadoAprobada()); }
  rechazar(orden) { orden._cambiarEstado(new EstadoRechazada()); }
  nombre() { return 'PRESUPUESTADA'; }
}
✅ Olor eliminado: Switch/if-else sobre tipo — 10 clases de estado (EstadoRecibida, EstadoEnDiagnostico, EstadoPresupuestada, etc.) implementan la misma interfaz y son perfectamente intercambiables. La OrdenServicio delega avanzar() al estado actual sin necesidad de if.
ISP Interface Segregation Principle — Segregación de Interfaces

Regla: Preferir interfaces pequeñas y específicas; ninguna clase debe implementar métodos que no usa.

Problema detectado: Si todos los Use Cases recibieran el TallerFacade completo (con 20+ métodos), cada caso de uso tendría acceso a funciones que no necesita.

❌ ANTES — Dependencia del Facade completo
// El UseCase recibe TODO el facade (violación ISP)
class CrearOrdenUseCase {
  constructor(facade) {
    this.facade = facade;
    // ❌ Tiene acceso a facade.login()
    // ❌ Tiene acceso a facade.generarFactura()
    // ❌ Tiene acceso a facade.eliminarOrden()
    // Solo necesita ordenRepo, clienteRepo,
    // vehiculoRepo y eventBus
  }
}
✅ DESPUÉS — Solo recibe lo que necesita
// CrearOrdenUseCase.js — Solo dependencias mínimas
class CrearOrdenUseCase {
  constructor(ordenRepo, clienteRepo,
             vehiculoRepo, eventBus) {
    this._ordenRepo    = ordenRepo;
    this._clienteRepo  = clienteRepo;
    this._vehiculoRepo = vehiculoRepo;
    this._eventBus     = eventBus;
    // ✅ Solo las 4 dependencias necesarias
    // ✅ No conoce facturas, usuarios ni login
  }
}
// GenerarFacturaUseCase — Diferente "interfaz"
class GenerarFacturaUseCase {
  constructor(ordenRepo, clienteRepo,
             facturaRepo, eventBus) { ... }
}
✅ Olor eliminado: Fat Interface — cada caso de uso recibe solo los repositorios que utiliza, no el facade entero con 20+ métodos. Esto reduce el acoplamiento y facilita el testing (solo se mockean las dependencias necesarias).
DIP Dependency Inversion Principle — Inversión de Dependencias

Regla: Depender de abstracciones, no de implementaciones concretas.

Problema detectado: Si los controladores instanciaran new PrismaClient() directamente, todo el código quedaría acoplado a Prisma (imposible cambiar a otra BD sin reescribir todo).

❌ ANTES — Acoplamiento directo
// Controlador crea sus propias dependencias
const { PrismaClient } = require('@prisma/client');
const prisma = new PrismaClient();

router.post('/', async (req, res) => {
  // ❌ Acoplado directamente a Prisma
  const orden = await prisma.orden.create({
    data: req.body
  });
  res.json(orden);
});
// ❌ Cambiar a MongoDB = reescribir TODO
// ❌ Imposible hacer tests unitarios
✅ DESPUÉS — container.js + abstracciones
// container.js — Wiring de dependencias
const ordenRepo =
  new PrismaOrdenRepository();
const clienteRepo =
  new PrismaClienteRepository();
const eventBus = new EventBus();

// El facade recibe abstracciones inyectadas
const tallerFacade = new TallerFacade({
  ordenRepo,    // ← abstracción
  clienteRepo,  // ← abstracción
  vehiculoRepo, // ← abstracción
  eventBus,     // ← abstracción
});
// ✅ Cambiar a MongoDB = solo nuevo repo
// ✅ Tests: inyectar repos en memoria
✅ Olor eliminado: Hard-coded dependency — el container.js actúa como contenedor de inyección de dependencias (DI manual). Las capas superiores (UseCase, Facade) reciben abstracciones; PrismaOrdenRepository extiende la clase abstracta OrdenRepository, que es la interfaz que la aplicación conoce.

🎨 Tema 2 — Mejoras Profesionales de UI/UX

Se implementaron 8 técnicas profesionales de interfaz (mínimo requerido: 6), cada una con justificación técnica.

1 Asistente / Wizard por Pasos — NuevaOrden.jsx

Problema de usabilidad

El formulario de "Nueva Orden" era una pantalla larga y saturada con todos los campos visibles simultáneamente (cliente, vehículo, prioridad, mecánico, servicios, notas). Esto genera sobrecarga cognitiva y errores de omisión.

Solución implementada

  • Wizard de 3 pasos: Paso 1 (Cliente y Vehículo), Paso 2 (Servicios), Paso 3 (Resumen y confirmación).
  • Barra de progreso visual con indicadores de paso completado ✅, activo 🔵 y pendiente ⚪.
  • Validación por etapa: no se puede avanzar sin seleccionar cliente y vehículo (Paso 1) o sin agregar al menos un servicio (Paso 2).
  • Resumen visual con tarjetas resumen antes de confirmar la creación.

Principio de diseño: Reducción de carga cognitiva (Progressive Disclosure) y prevención de errores (validación por etapa).

Archivo: frontend/src/pages/NuevaOrden.jsx

2 Menú Lateral Colapsable — Layout.jsx

Problema de usabilidad

El sidebar de navegación ocupaba ~256px de ancho fijo, reduciendo el espacio disponible para el contenido principal en pantallas medianas.

Solución implementada

  • Botón toggle para colapsar/expandir el sidebar en desktop.
  • Modo colapsado: sidebar de 72px mostrando solo íconos (sin texto).
  • Tooltips al pasar el mouse sobre cada ítem al estar colapsado (accesibilidad).
  • Persistencia del estado en localStorage (el sidebar recuerda si estaba colapsado).
  • Transición suave con transition-all duration-300.

Principio de diseño: Flexibilidad y eficiencia de uso (Heurística de Nielsen #7).

Archivo: frontend/src/components/Layout.jsx

4 DataGrid Profesional — Ordenes.jsx

Problema de usabilidad

La lista de órdenes era un listado simple sin encabezados, sin paginación, y con todas las órdenes visibles en un solo scroll largo.

Solución implementada

  • Encabezados de columna fijos (Cliente, Estado, Fecha) con indicadores de ordenamiento.
  • Filas alternadas (zebra striping) para mejorar la legibilidad horizontal.
  • Ordenamiento por columna — clic en el encabezado para ordenar ascendente/descendente.
  • Paginación local — 10 registros por página con navegación numérica.
  • Tooltips con ID en cada fila al hacer hover.
  • Formato condicional — badges de estado con colores según el ciclo de vida (recibida=gris, en reparación=azul, facturada=verde, rechazada=rojo).

Principio de diseño: Consistencia y estándares (Heurística de Nielsen #4) — las tablas profesionales siguen convenciones establecidas.

Archivo: frontend/src/pages/Ordenes.jsx

5 Ocultación de Opciones por Rol/Permiso — navigationConfig.js

Problema de usabilidad

Mostrar funciones que el usuario no puede usar genera confusión y riesgo de seguridad.

Solución implementada

  • Cada ítem de navegación tiene un arreglo roles que define quién lo puede ver.
  • La función getNavItemsByRol(rol) filtra los ítems visibles.
  • ADMIN ve todo (Dashboard, Órdenes, Clientes, Usuarios, Equipo Técnico, Facturación).
  • MECANICO solo ve Dashboard y Órdenes.
  • CAJERO ve Dashboard, Órdenes, Clientes y Facturación.
  • El botón "Eliminar Orden" solo aparece si user.rol === 'ADMIN'.

6 Retroalimentación Visual — Toast + ConfirmModal + Spinners

Problema de usabilidad

window.alert() y window.confirm() bloquean el hilo principal de JavaScript, tienen un diseño inconsistente entre navegadores, y rompen el flujo del usuario.

Solución implementada

  • Toast.jsx — notificaciones flotantes animadas (slide-in) que desaparecen automáticamente. Tres variantes: success (verde), error (rojo), info (azul).
  • ConfirmModal.jsx — diálogos de confirmación estilizados con opción de peligro (botón rojo) para acciones destructivas.
  • Spinners de carga — indicadores de loading animados con animate-spin en botones de submit.
  • Badges de estado — etiquetas de color codificado para cada estado de la orden.

Principio de diseño: Visibilidad del estado del sistema (Heurística de Nielsen #1).

Archivos: Toast.jsx, ConfirmModal.jsx

7 Ventanas Modales (Modal / Dialog)

Problema de usabilidad

Las acciones de confirmación (eliminar orden, generar factura) redirigían o abrían diálogos nativos, causando pérdida de contexto.

Solución implementada

  • ConfirmModal — se usa para confirmar la generación de factura y la eliminación de órdenes.
  • Overlay oscuro que centra la atención del usuario en la decisión.
  • Botones con semántica de color (rojo para destruir, azul para confirmar).
  • Soporte para modo "peligro" con estilos diferenciados.

Principio de diseño: Prevención de errores — pedir confirmación explícita antes de acciones irreversibles.

Archivos: ConfirmModal.jsx, Ordenes.jsx, Facturacion.jsx

8 Barra de Menú Superior (Header) con Título Dinámico

Problema de usabilidad

Sin indicación clara de la sección activa, el usuario puede perderse dentro del sistema.

Solución implementada

  • Header superior fijo con backdrop-blur semitransparente que muestra el título dinámico de la página actual.
  • Botón hamburguesa en móvil para abrir el sidebar.
  • Título con gradiente de color que refuerza la identidad de marca.

Principio de diseño: Reconocimiento antes que recuerdo (Nielsen #6) — el header le dice al usuario dónde está.

Archivo: frontend/src/components/Layout.jsx (líneas 164-177)
